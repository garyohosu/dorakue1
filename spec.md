# dorakue1 仕様書

この文書は、Quora回答用の検証プロジェクト `dorakue1` のゲーム仕様書です。次の段階で Codex Game Studio、Codex CLI、Claude Code、ImageGen、Playwright を使って段階的に実装できるよう、ゲーム内容、データ、素材、テスト、制作ログの方針をまとめます。

このタスクでは実装しません。画像生成、音声生成、テスト実行もまだ行いません。作成対象はこの `spec.md` のみです。

## 0. 仮タイトル案と採用タイトル

候補:

| 案 | タイトル | 英語サブタイトル | 評価 |
|---|---|---|---|
| A | 暁の小径 | Path of Dawn | 短く、旅と朝の回復を連想しやすい |
| B | 星霧の灯 | Lantern of Star Mist | 幻想的だが少し抽象的 |
| C | 白鐘の旅人 | The White Bell Wanderer | 王国名と結びつくがやや説明的 |

採用タイトル:

**暁の小径 / Path of Dawn**

採用理由:

- 短く覚えやすい。
- 1980年代風RPGらしい素朴な響きがある。
- 既存有名RPGの固有名詞に似すぎない。
- 「長い夜を終わらせ、朝を取り戻す」という物語と合う。

## 1. プロジェクト概要

### 1.1 このゲームの目的

`dorakue1` は、Quoraの質問「現在、上手にAIを駆使すれば、ドラクエ1くらいのゲームはプログラミングを知らなくても作れてしまうのでしょうか？」に対して、実際にAIコーディングツールを使って検証するためのブラウザゲームです。

このプロジェクトの主目的は、単に「作れそう」と説明することではありません。Codex Game Studio、Codex CLI、Claude Code、ImageGen、Playwrightによるブラウザ確認を組み合わせ、非プログラマが段階的な自然言語指示だけで、1980年代の古典的な一人旅RPG相当のゲームを完成に近づけられるかを実証します。

### 1.2 Quora回答用の実験であること

完成物そのものに加えて、制作過程も重要な成果物です。

- どのAIツールを使ったか。
- どんな指示を出したか。
- どこまで一発で動いたか。
- どこでバグが出たか。
- 非プログラマでも判断できたことは何か。
- プログラミング知識が必要になった場面は何か。
- 最終的に「作れてしまう」と言えるのか。

これらを後で `.logs/` に記録し、Quora回答の材料にします。このタスクでは `.logs/` は作成しません。

### 1.3 AIコーディングツールで検証すること

検証対象:

- 仕様書をAIが実装可能な粒度に落とせるか。
- 非プログラマが「Phase 1だけ」「次は戦闘だけ」のように小さく依頼できるか。
- AIがコードを生成し、修正し、レビューし、テストまで回せるか。
- 画像や音声などのゲーム素材もAI生成で揃えられるか。
- ブラウザ確認をPlaywrightで自動化できるか。

### 1.4 ブラウザゲームにする理由

- URLを開くだけで遊べる。
- GitHub Pagesで公開しやすい。
- Windows、macOS、Linuxの差が少ない。
- スクリーンショットやエラー内容をAIに渡しやすい。
- Playwrightで自動テストしやすい。
- 非プログラマでも「ブラウザで表示されたか」を判断しやすい。

### 1.5 GitHub Pagesで公開する前提

最終的にはGitHub Pagesで公開します。理想の公開方式は、Viteでビルドした `dist/` をGitHub Pagesへ配信する形です。

初期段階ではローカル開発サーバーで確認し、公開段階でGitHub Pages用の設定を追加します。

### 1.6 完成時の想定プレイ時間

- 最小完成版: 30分から60分。
- 理想完成版: 2時間から4時間。
- テスト用の高速クリアルート: 15分から30分。

### 1.7 最小完成版と理想完成版の違い

最小完成版は「ゲームとして最後まで遊べる」ことを優先します。画像、BGM、演出が未完成でも、移動、会話、戦闘、成長、重要アイテム、ラスボス、エンディング、セーブ・ロードが動けば成功です。

理想完成版は、画像素材、BGM、効果音、歩行アニメ、複数ダンジョン、中ボス、Playwrightテスト、制作ログまで揃えた公開可能版です。

### 1.8 「ドラクエ1相当」の意味

この仕様書でいう「ドラクエ1相当」は、既存作品の再現、コピー、クローンではありません。以下のような古典的一人旅RPGとしての構造を意味します。

- 一人旅RPGとして完結している。
- ワールドマップがある。
- 城、町、洞窟、塔、ラストダンジョンがある。
- NPC会話でヒントを得る。
- 装備更新で強くなる。
- レベル上げがある。
- 重要アイテムで進行する。
- ラスボスを倒すとエンディングになる。
- 2から4時間程度でクリアできる小規模RPGとして成立している。

### 1.9 権利上の注意

このゲームは完全オリジナル作品です。以下は禁止します。

- 既存有名RPGの固有名詞。
- 既存作品のモンスター名。
- 既存作品の呪文名。
- 既存作品の音楽、旋律、効果音の模倣。
- 既存作品のキャラクターデザインの模倣。
- 既存作品のマップ構造の再現。
- 既存作品のセリフの引用。
- 既存作品のUIをそのまま再現すること。

仕様上は「古典的JRPG風」「1980年代風RPG」「一人旅RPG」「コマンド式戦闘」のような一般表現を使って構いません。

## 2. 開発方針

### 2.1 Codex Game Studioの役割

Codex Game Studioが利用できる場合は、ゲーム構造、画面遷移、シーン分割、データ設計、実装計画の整理に使います。

非プログラマ向けに言えば、Codex Game Studioは「ゲーム全体をどういう部品に分けて作るかを相談する相手」です。いきなりコードを書くのではなく、ゲームの骨組み、必要な画面、実装順序、テスト観点を整理する役割にします。

このセッションで専用ツールが利用できない場合でも、仕様書上は「Codex Game Studio向けの依頼文」を残します。

### 2.2 Codex CLIの役割

Codex CLIは、実際のコード作成、修正、ファイル編集、ローカル実行、テスト修正を担当します。

非プログラマ向けに言えば、Codex CLIは「リポジトリの中で手を動かしてくれる実装担当」です。各Phaseごとに、変更するファイル、完了条件、確認方法を指定して依頼します。

### 2.3 Claude Codeの役割

Claude Codeは、別視点でのレビュー、仕様の抜け確認、コードの読みやすさ確認、リファクタ提案に使います。

非プログラマ向けには、「Codexが作ったものを別のAIにチェックしてもらう」役割です。実装を変えずにレビューだけ依頼するフェーズを用意します。

### 2.4 ImageGenの役割

ImageGenは、キャラクター、敵、タイル、UI素材、タイトルロゴ、背景素材の生成に使います。

ただし初期実装では、色付き四角や仮画像で進めます。ゲームが最後まで動くようになってから、ImageGenで生成した素材へ置き換えます。

素材生成時は、`.prompts/` にプロンプトを保存する方針です。このタスクでは `.prompts/` は作成しません。

### 2.5 Playwrightの役割

Playwrightは、ブラウザ上でゲームが動いているかを確認するために使います。

非プログラマ向けには、「人間が毎回全部を手で確認しなくても、ブラウザを自動操作して基本動作を検査してくれる道具」です。

確認対象:

- タイトル画面が表示されるか。
- 新規開始できるか。
- 主人公が移動できるか。
- 戦闘が始まるか。
- メニューが開くか。
- セーブ・ロードできるか。
- BGM/SEが鳴るか。

### 2.6 人間ユーザーの役割

人間ユーザーは、最終的な面白さ、分かりやすさ、違和感、権利上の危うさを判断します。

人間が見るべきこと:

- 次に行く場所が分かるか。
- 戦闘が退屈すぎないか。
- 敵が強すぎないか。
- 店で装備を買う意味があるか。
- 既存作品に似すぎていないか。
- Quora回答として説得力があるか。

## 3. 技術方針

### 3.1 候補比較

| 候補 | 構成 | 長所 | 短所 |
|---|---|---|---|
| A | HTML / CSS / JavaScript / Canvas API / 追加ライブラリなし | 依存が少ない。初心者に説明しやすい | タイル、シーン、音声、当たり判定、アニメを自前実装する必要がある |
| B | HTML / CSS / JavaScript / Phaser / Vite / GitHub Pages | 2Dゲーム、タイル、画像、音声、入力、シーン管理に強い。JSなのでTSより導入しやすい | Node.jsとnpmが必要 |
| C | TypeScript / Phaser / Vite / GitHub Pages | 型安全で大規模化に強い | 非プログラマには型エラーが難しく、AIへの依頼もやや複雑 |

### 3.2 採用構成

**候補B: HTML / CSS / JavaScript / Phaser / Vite / GitHub Pages** を採用します。

理由:

- 2DタイルRPGに必要な機能をPhaserが持っている。
- シーン分割により、タイトル、フィールド、戦闘、メニューを整理しやすい。
- 画像や音声を扱いやすい。
- Playwrightでローカル開発サーバーを開いて確認しやすい。
- GitHub Pagesへ公開しやすい。
- TypeScriptよりJavaScriptの方が非プログラマに説明しやすい。
- Codex CLIやClaude Codeに「このSceneだけ直して」と依頼しやすい。

### 3.3 技術スタック

- 実装言語: JavaScript
- ゲームエンジン: Phaser
- 開発サーバー・ビルド: Vite
- 描画: PhaserのCanvas/WebGL描画
- 音声: Phaser Sound + 必要に応じてWeb Audio API
- セーブ: LocalStorage
- テスト: Playwright
- 公開: GitHub Pages

### 3.4 初期実装での注意

- Phase 1までは雛形だけにする。
- Phase 2からタイトル画面を作る。
- 画像素材がない間は、単色タイルと簡易矩形で代用する。
- BGM/SEは後半Phaseまで入れない。
- 1ファイルに詰め込みすぎず、`src/game/`、`src/data/`、`src/scenes/` に分ける。

## 4. ゲーム全体コンセプト

### 4.1 世界名

`リュミナ地方`

古くから「夜明けの鐘」に守られてきた小さな地方です。山、森、海辺、古塔、黒い岬があり、短い冒険に必要な地形を一通り持ちます。

### 4.2 王国名

`セリオル王国`

白い鐘楼を中心に栄えた小国です。国土は小さいが、古い灯火の儀式を守っています。

### 4.3 主人公名

`アレン`

名前変更機能は初期版では不要です。拡張版で追加可能にします。

### 4.4 主人公の立場

主人公は、王都の外れで灯台の火を守る家に生まれた若い `灯守見習い` です。騎士ではありませんが、古い灯火の儀式に関わる血筋を持ちます。

### 4.5 敵の親玉

`黒冠卿オルヴェス`

かつて王国の星読みだった人物です。禁じられた夜の術に魅入られ、朝を遅らせる黒い霧を広げています。

### 4.6 敵勢力の名前

`夜冠衆`

黒冠卿オルヴェスに従う影、古い鎧、黒霧の術士たちの総称です。

### 4.7 世界に起きている異変

- 夜が通常より長く続く。
- 朝になっても空が薄暗い。
- 森や洞窟に見慣れない魔物が出る。
- 古い塔の頂上に青い火が灯った。
- 海辺の祠に続く道が黒い霧で閉ざされた。
- 王城の白鐘が鳴らなくなった。

### 4.8 旅の目的

王から使命を受けた主人公は、各地に残された重要アイテムを集め、海辺の祠で黒い岬への道を開き、黒冠砦で黒冠卿オルヴェスを倒します。

### 4.9 ラスボスを倒すために必要な重要アイテム

最終的に `暁の印` を完成させる必要があります。`暁の印` は単独アイテムではなく、以下の重要アイテムが揃った状態を表す進行フラグです。

- `月紋の鍵`: 古い塔の封印を開く。
- `青灯の珠`: 王城の白鐘を一度だけ鳴らす力を持つ。
- `朝霧の鏡`: 黒い霧の道を照らす。
- `潮路の貝`: 海辺の祠で道を開く。
- `白鐘の破片`: 王城の宝物庫で得る証。
- `暁の印`: 上記を揃え、海辺の祠で完成する最終進行フラグ。

### 4.10 冒険の流れ

1. タイトル画面から新しく始める。
2. 白鐘城で王から使命を受ける。
3. 始まりの町リノンで情報と装備を整える。
4. 森の洞窟で `月紋の鍵` を入手する。
5. 山の村カラムで古い塔の情報を得る。
6. 古い塔で中ボスを倒し、`青灯の珠` を入手する。
7. 白鐘城へ戻り、王から `朝霧の鏡` と `白鐘の破片` を受け取る。
8. 海辺の祠で `潮路の貝` を使い、黒冠砦への道を開く。
9. 黒冠砦でラスボスを倒す。
10. エンディング。

### 4.11 エンディングの内容

黒冠卿オルヴェスを倒すと、長く続いた夜が終わります。白鐘城の鐘が鳴り、リュミナ地方に朝日が戻ります。主人公は大げさな英雄ではなく、灯守として新しい朝を見届ける存在として描きます。

最後に、Quora実験として「このゲームはAIコーディングツールを使った制作検証として作られた完全オリジナル作品です」という注記を表示します。

## 5. ゲーム進行フロー

### 5.1 タイトル画面

プレイヤーの目的:

- 新しく始める、続きから、操作説明、クレジットを選ぶ。

必要なイベント:

- セーブデータがあるかを確認する。

必要なアイテム:

- なし。

推奨レベル:

- なし。

出現する敵:

- なし。

NPCヒント:

- なし。

詰まりやすい点:

- セーブデータがないのに続きからを選んだ場合。

実装上の注意:

- `LocalStorage` のキー存在確認を行う。
- 操作説明は簡潔に表示する。

### 5.2 新しく始める

プレイヤーの目的:

- 初期状態でゲームを開始する。

必要なイベント:

- プレイヤーデータ初期化。
- 初期マップを白鐘城に設定。

必要なアイテム:

- 初期装備 `木彫りの短剣`、`旅布の上着`
- `小癒し草` x 2
- 80リム

推奨レベル:

- 1

出現する敵:

- なし。

NPCヒント:

- 王と話すよう誘導する兵士。

詰まりやすい点:

- 何をすればよいか分からない。

実装上の注意:

- 開始直後、王の前に主人公を配置する。

### 5.3 王城で使命を受ける

プレイヤーの目的:

- セリオル王と会話し、旅の目的を知る。

必要なイベント:

- `acceptedQuest = true`

必要なアイテム:

- なし。

推奨レベル:

- 1

出現する敵:

- なし。

NPCヒント:

- 兵士が「南の町で準備を」と伝える。

詰まりやすい点:

- 王と話さずに城を出られてもよいが、町NPCが王と話すよう促す。

実装上の注意:

- 王のセリフは進行フラグで変える。
- 城内にセーブ係を置く。

### 5.4 始まりの町で情報を集める

プレイヤーの目的:

- 装備と回復道具を買う。
- 森の洞窟の場所を知る。

必要なイベント:

- 老人NPCから `learnedCaveHint = true`

必要なアイテム:

- 必須ではないが、`小癒し草`、`銅打ちの剣` 推奨。

推奨レベル:

- 1から2

出現する敵:

- 町内はなし。町外で `風切り草`、`小さな泥人`。

NPCヒント:

- 「北東の森に、月の紋が刻まれた洞がある」

詰まりやすい点:

- 装備を買わずに洞窟へ行くと苦しい。

実装上の注意:

- 店UIは最初は簡単でよい。
- 宿屋でHP/MP全回復できるようにする。

### 5.5 森の洞窟で重要アイテムを入手

プレイヤーの目的:

- 森の洞窟の奥で `月紋の鍵` を入手する。

必要なイベント:

- 宝箱を開ける。
- `gotMoonKey = true`

必要なアイテム:

- 推奨: `小癒し草` x 3

推奨レベル:

- 2から4

出現する敵:

- `小さな泥人`
- `苔羽虫`
- `洞火の影`
- `木根の獣`

NPCヒント:

- 町人が「洞窟では帰り道を見失うな」と言う。

詰まりやすい点:

- 回復道具不足。
- ダンジョンからの戻り方が分かりにくい。

実装上の注意:

- 最初の洞窟は1階層でよい。
- 出口へ戻る導線を短めにする。

### 5.6 山の村で次の目的地を知る

プレイヤーの目的:

- 古い塔へ行く条件を知る。

必要なイベント:

- 村長と会話。
- `learnedTowerHint = true`

必要なアイテム:

- `月紋の鍵`

推奨レベル:

- 4から5

出現する敵:

- 村内はなし。周辺に `灰毛の山犬`、`石殻兵`。

NPCヒント:

- 「月紋の鍵を持つ者だけが塔の扉を開けられる」

詰まりやすい点:

- `月紋の鍵` を取らずに来た場合。

実装上の注意:

- 鍵がない場合、塔入口で明確なメッセージを出す。

### 5.7 古い塔で封印を解くアイテムを入手

プレイヤーの目的:

- 塔の上階で中ボスを倒し、`青灯の珠` を入手する。

必要なイベント:

- `defeatedTowerBoss = true`
- `gotBlueLampOrb = true`

必要なアイテム:

- `月紋の鍵`

推奨レベル:

- 6から8

出現する敵:

- `錆びた操り人形`
- `塔の衛兵影`
- `黒霧の術士`
- `羽音の監視者`

NPCヒント:

- 山の村の少女が「塔の上で青い火を見た」と言う。

詰まりやすい点:

## 7. 操作仕様

### 7.1 キーボード操作

| キー | フィールド | メニュー | 戦闘 |
|---|---|---|---|
| 方向キー | 移動 | カーソル移動 | コマンド選択 |
| WASD | 移動 | カーソル移動 | コマンド選択 |
| Enter | 決定、会話送り | 決定 | 決定 |
| Space | 決定、会話送り | 決定 | 決定 |
| Z | 決定 | 決定 | 決定 |
| Esc | キャンセル | 閉じる | 戻る |
| X | メニュー、キャンセル | 閉じる | 戻る |
| M | メニュー | 閉じる | 使用しない |

### 7.2 長押し移動

- 初期版では長押し移動あり。
- 1マス移動後、150msから220ms程度の間隔で連続移動。
- メニュー中はキーリピートを遅めにし、誤操作を防ぐ。

### 7.3 1マス移動の速度

- 1タイル32px。
- 歩行アニメなしの最小版: 即時移動。
- 理想版: 120msから160msで1マス分スライド。

### 7.4 戦闘中のコマンド選択

- 上下でコマンド移動。
- Enter/Space/Zで決定。
- Esc/Xで前の選択に戻る。
- 道具や術の一覧では、使用不可項目を灰色表示。

### 7.5 スマホ対応

初期版ではスマホ完全対応は対象外です。

将来案:

- 左下に仮想十字キー。
- 右下にA/Bボタン。
- 右上にメニューボタン。
- 画面比率は16:9または4:3を維持。

## 8. マップ仕様

### 8.1 基本

- タイルベース。
- 1タイルサイズ: 32px。
- 内部座標はタイル単位。
- 表示範囲は 20 x 15 タイル程度を推奨。
- ワールドマップはスクロールあり。
- 建物内や小マップは固定カメラでもよい。

### 8.2 主人公表示

- ワールドマップでは主人公を画面中央付近に固定し、マップをスクロール。
- 端ではスクロールを止め、主人公が画面端へ寄る。

### 8.3 タイル種別

| タイル | 通行 | エンカウント | 用途 |
|---|---|---|---|
| 草原 | 可 | 可 | 基本地形 |
| 森 | 可 | 可、高め | 洞窟周辺 |
| 山 | 不可 | 不可 | 境界 |
| 水 | 不可 | 不可 | 海、湖 |
| 危険湿地 | 可 | 可、高め。歩行時ダメージ候補 | 終盤地形 |
| 橋 | 可 | 可または不可 | 地域接続 |
| 道 | 可 | 可、低め | 安全な移動路 |
| 砂地 | 可 | 可 | 海辺 |
| 城 | 入口 | 不可 | 王城 |
| 町 | 入口 | 不可 | 始まりの町 |
| 洞窟 | 入口 | 不可 | 森の洞窟 |
| 塔 | 入口 | 不可 | 古い塔 |
| 祠 | 入口 | 不可 | 海辺の祠 |
| 床 | 可 | 場所による | 屋内 |
| 壁 | 不可 | 不可 | 屋内境界 |
| 階段 | 可 | 不可 | 階層移動 |
| 扉 | 条件次第 | 不可 | フラグで開閉 |
| 宝箱 | 不可 | 不可 | 調べる対象 |

### 8.4 イベント発火条件

- 入口: そのタイルに乗る。
- NPC会話: NPCの隣でNPC方向を向き、決定。
- 宝箱: 隣接して決定、または `周囲を見る`。
- 扉: 正面で決定。条件を満たせば開く。
- 階段: そのタイルに乗る。
- 固定敵: 指定タイルに乗る、または会話する。

### 8.5 宝箱

- 宝箱IDで開封状態を管理する。
- 開封済みなら `からっぽだ。` と表示。
- セーブ対象にする。

### 8.6 扉

- 扉は `requiredFlag` または `requiredItemId` を持つ。
- 条件不足時は不足理由をメッセージで出す。

### 14.3 戦闘

ターン開始:

- 状態異常の継続処理。
- 毒ならHPを少し減らす。
- 眠りなら行動不能判定。

コマンド選択:

- `攻める`
- `術法`
- `道具`
- `退く`

素早さによる先攻判定:

```text
主人公先攻確率 = 50 + (主人公素早さ - 敵素早さ) * 3
下限 = 25%
上限 = 85%
```

攻撃処理:

- 通常攻撃ダメージを計算。
- 5%の確率で `冴えた一撃` が発生。

術処理:

- MP不足なら使用不可。
- 攻撃、回復、補助、脱出、帰還を術ごとに処理。

道具処理:

- 所持数を確認。
- 使用後に所持数を減らす。

敵行動:

- 通常攻撃。
- 特殊行動。
- HP条件による行動変化。

勝利判定:

- 敵HPが0以下。
- 経験値と所持金を加算。
- レベルアップ判定。

敗北判定:

- 主人公HPが0以下。
- ゲームオーバー画面へ。

逃走判定:

```text
逃走成功率 = 55 + (主人公素早さ - 敵素早さ) * 4
下限 = 20%
上限 = 90%
```

ボス戦:

- 逃走不可。

戦闘ログ:

- 1行から3行程度。
- テキスト送りあり。

### 14.4 ダメージ計算式

通常攻撃:

```text
基礎ダメージ = 攻撃側攻撃力 - floor(防御側防御力 / 2)
乱数補正 = 0.85 から 1.15
最終ダメージ = max(1, floor(基礎ダメージ * 乱数補正))
```

冴えた一撃:

```text
発生率 = 5%
ダメージ = 通常攻撃ダメージ * 1.8
```

敵の特殊攻撃:

```text
特殊ダメージ = 基本威力 + floor(敵攻撃力 / 3) - floor(主人公防御力 / 4)
最低値 = 3
```

術ダメージ:

```text
術ダメージ = 術基本威力 + floor(使用者レベル * 1.5) + 乱数(-3 から +3)
```

回復量:

```text
回復量 = 術基本回復量 + floor(使用者レベル * 2) + 乱数(-3 から +3)
```

- 購入時: 装備するか確認。
- 店員セリフ: `身を守る物は、旅の足を止めないための道具です。`

- 売るもの: 回復、帰還、補助道具。
- 最大所持数超過時: `これ以上は持てません。`
- 店員セリフ: `小癒し草は多めにどうぞ。洞では一枚の葉が命をつなぎます。`

宿屋:

- 料金: 町ごとに設定。
- 処理: HP/MP全回復、毒解除、最後に休んだ場所更新。
- セリフ: `今夜は安全な屋根の下でお休みください。`

特殊な情報屋:

- カラム山村に配置。
- 価格: 30リム。
- 効果: 危険地帯や隠し宝箱のヒント。
- セリフ: `山道の噂は安くない。聞くかい。`

### 14.12 セーブ・ロード

セーブ方式:

- LocalStorage。

キー:

- `path_of_dawn_save_v1`

スロット数:

- 最小版: 1枠。
- 理想版: 3枠も検討可。

保存内容:

- プレイヤーデータ。
- 現在マップID。
- 座標。
- 向き。
- 所持アイテム。
- 装備。
- 重要アイテム。
- 進行フラグ。
- 開封済み宝箱。
- 撃破済み固定敵。
- 音量設定。
- 最後に休んだ場所。

タイトル画面からのロード:

- セーブデータがあれば `続きから` を有効化。

セーブ失敗時:

- `記録に失敗しました。ブラウザの保存設定を確認してください。`

バージョン違い:

- `version` を保存。
- 読み込み時に不明バージョンなら警告を出し、タイトルへ戻す。

削除方法:

- 設定画面に `記録を消す` を追加する。
- 初期版ではブラウザ開発者ツールでLocalStorage削除でも可。READMEに説明する。

## 18. UI・メッセージ仕様

### 18.1 ウィンドウデザイン

- 黒または濃紺の背景。
- 白または薄いクリーム色の文字。
- 枠は白、薄青、または金を少量。
- 角丸にしすぎない。
- 既存作品のUIをそのまま模倣しない。

### 18.2 フォント方針

- 初期版: CSSの `monospace` または読みやすいシステムフォント。
- 理想版: ドット風Webフォントも検討。ただしライセンス確認必須。
- 日本語が読みにくいフォントは避ける。

### 18.3 文字サイズ

- 通常メッセージ: 16pxから20px。
- ステータス小窓: 14pxから16px。
- タイトル: 48px以上。

### 18.4 メッセージ表示速度

- 1文字あたり20msから35ms。
- 決定キーで全文表示。
- 全文表示後の決定キーで次へ。

### 18.5 カーソル

- `>` または小さな三角アイコン。
- 点滅は任意。
- 効果音は小さく。

### 18.6 表示要素

- HP/MP表示: `HP 24/24  MP 6/6`
- 所持金: `80リム`
- 戦闘ログ: 下部メッセージウィンドウ。
- 店画面: 商品名、価格、所持数、装備中比較。
- セーブ確認: `記録しますか？ はい / いいえ`

### 18.7 メッセージ例

| 場面 | メッセージ |
|---|---|
| 攻撃時 | アレンの一撃。 |
| ダメージ時 | 風切り草に 8 の傷を与えた。 |
| 敵攻撃 | 小さな泥人の体当たり。 |
| 敵撃破 | 風切り草をしりぞけた。 |
| 経験値 | 4の経験を得た。5リムを手に入れた。 |
| レベルアップ | アレンは灯守として少し強くなった。 |
| 術習得 | 新しい術法「癒しの息」を覚えた。 |
| アイテム入手 | 月紋の鍵を手に入れた。 |
| 所持金不足 | リムが足りないようだ。 |

| セーブ完了 | 旅の記録を残した。 |

## 19. バランス設計

### 19.1 クリア想定レベル

- 最小完成版: レベル10から12。
- 理想完成版: レベル14から16。

### 19.2 エリア推奨レベル

| エリア | 推奨Lv | 備考 |
|---|---:|---|
| 王城 | 1 | 戦闘なし |
| リノン周辺 | 1-2 | 操作練習 |
| 森の洞窟 | 2-4 | 初装備が重要 |
| 山道 | 4-6 | 防御不足だと苦しい |
| 星見の塔 | 6-8 | 中ボス |
| 海辺の祠周辺 | 8-10 | 強敵混在 |
| 危険地帯 | 9-13 | レベル上げ用 |
| 黒冠砦 | 10-15 | 終盤 |
| ラスボス | 14-16 | 装備と回復が重要 |

### 19.3 敵の強さ

- 序盤: 2から4ターンで倒せる。
- 中盤: 装備更新がないと被ダメージが重い。
- 終盤: 回復術と道具を使う前提。
- 危険地帯: 経験値は高いが消耗も大きい。

### 19.4 経験値カーブ

- レベル1から5は短時間で上がる。
- レベル6から10はダンジョン攻略で自然に上がる。
- レベル11以降は少し稼ぎが必要。

### 19.5 所持金カーブ

- 序盤は武器か防具のどちらを先に買うか迷う程度。
- 中盤は洞窟や塔の宝箱で装備更新を助ける。
- 終盤は最強装備を宝箱で入手し、金不足で詰まないようにする。

### 19.6 装備更新タイミング

- リノン到着: 銅打ちの剣または革編みの胴衣。
- 洞窟後: 鉄葉の剣、鎖糸の外套。
- 塔前: 銀星の剣、銀糸の鎧。
- 砦内: 霧断ち、霧よけの衣。
- ラスボス前: 暁の刃、暁の鎧。

### 19.7 価格

- リノン宿屋: 8リム。
- カラム宿屋: 24リム。
- 終盤簡易宿: 50リム。
- 小癒し草: 12リム。
- 大癒し草: 60リム。

### 19.8 テスト時の確認項目

- レベル1で森に直行すると危険だが即死しすぎない。
- レベル4で洞窟をクリアできる。
- レベル8で塔中ボスに勝てる。
- レベル14、標準装備でラスボスに勝機がある。
- 回復道具を買えば安定する。
- 逃走成功率が高すぎない。

```text
spec.mdを読んで、Phase 1だけ実装してください。
変更したファイル一覧、起動コマンド、確認手順、次のPhaseの提案を最後に表示してください。
```

### Claude Code向け

```text
spec.mdを読んで、現在の実装が仕様に沿っているかレビューしてください。
実装は変更せず、問題点、抜け、次に修正すべき点を箇条書きで出してください。
```

### ImageGen向け

```text
spec.mdの画像アセット仕様を読み、主人公スプライト、通常敵、タイルセット、UI素材の生成プロンプトを .prompts/ に整理してください。
まだ画像生成はしないでください。
```

### Playwright向け

```text
現在のゲームをブラウザで起動し、タイトル画面、移動、戦闘、メニュー、セーブ、BGM/SEの基本動作を確認してください。
失敗した項目を .logs/ に記録してください。
```

## 27. 非プログラマ向け作業ルール

- 1回のAI指示で大きく作りすぎない。
- 各フェーズごとに動作確認する。
- エラーが出たらエラーメッセージをそのままAIに渡す。
- 「何をしたら」「どうなった」を具体的に書く。
- 仕様変更は先に `spec.md` に追記する。
- 完成より先に「動く最小版」を優先する。
- 画面が地味でも、まず遊べることを重視する。
- うまく動いたらGitにコミットする。
- 画像や音声は後回しにしてよい。
- 既存作品に似すぎていないか毎回確認する。

## 28. 受け入れ基準

最終的に以下を満たせば、古典的一人旅RPG相当のブラウザゲームをAIで作れたと判断します。

- ブラウザで起動する。
- GitHub Pagesで公開できる。
- タイトル画面がある。
- 主人公を移動できる。
- ワールドマップがある。
- 城、町、洞窟、塔、祠、ラストダンジョンを移動できる。
- NPCと会話できる。
- 会話から次の目的地が分かる。
- ランダム戦闘がある。
- 敵を倒すと経験値と所持金が入る。
- レベルアップする。
- 店で買い物できる。
- 宿屋で回復できる。
- 装備を変更できる。
- 道具を使える。
- 術を使える。
- 重要アイテムでイベントが進む。
- 中ボスまたは固定敵がいる。
- ラスボスを倒すとエンディングになる。
- セーブ・ロードできる。
- Playwrightで主要動作を確認できる。
- 既存作品の固有名詞、音楽、キャラクター、セリフを使っていない。

## 29. 除外するもの

初期版では以下を対象外にします。

- スマホ完全対応。
- 複数パーティ。
- 複雑な職業システム。
- 3D。
- ネットワーク機能。
- 課金要素。
- オンラインランキング。
- 大量のサブクエスト。
- 複数エンディング。
- 既存作品の再現。

## 30. 実装担当AIへの最終注意

- まずPhase 1だけを実装してください。
- 仕様書にない大きな変更をしないでください。
- 画像、音声、テストは段階的に追加してください。
- 実装時は、毎回 `spec.md` の該当章を参照してください。
- 既存有名RPGに似た名前、音、見た目、台詞を避けてください。
- 非プログラマが追えるよう、変更内容と確認手順を短く報告してください。
